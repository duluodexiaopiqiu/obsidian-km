# 飞书知识库 ↔ 本地 Obsidian 同步方案

## 核心原则

**以飞书内容为准**：同步方向为单向（飞书 → 本地），每次同步时用飞书文档内容覆盖本地文件。

## 目录组织方案

有子文档的知识库节点建文件夹，没有子文档的保持平铺。

**编号规则：**
- `XX-标题.md` → 主文档（XX 是飞书章节顺序）
- `XX-YY-标题.md` → 子文档（XX 对应主文档编号，YY 是子文档序号）
- 有子文档的自动加 `📁 XX-主题/` 文件夹

```
📁 知识库根目录/
├── 📄 00-大纲.md                     ← 主文档（无子文档，平铺）
├── 📄 01-主文档.md
├── 📄 02-主文档.md
├── 📁 03-主题A/                      ← 主文档有子文档
│   ├── 📄 03-主题A.md                ← 主文档
│   ├── 📄 03-01-子文档1.md           ← 子文档
│   └── 📄 03-02-子文档2.md
├── 📁 04-主题B/
│   ├── 📄 04-主题B.md
│   └── 📄 04-01-子文档.md
└── .feishu-mapping.json             ← 映射文件
```

## 映射文件 `.feishu-mapping.json`

每个本地文件与飞书文档的对应关系：

```json
{
  "version": "1.0",
  "space_id": "飞书知识空间ID",
  "space_name": "知识空间名称",
  "updated_at": "最后更新时间",
  "files": [
    {
      "local": "相对路径/文件名.md",
      "title": "飞书文档标题",
      "obj_token": "文档内容token（用于 docs +fetch）",
      "node_token": "知识库节点token（用于 wiki 操作）",
      "parent_node_token": "父节点token（空=根节点）",
      "has_child": true
    }
  ]
}
```

> **为什么需要两个 token？**
> - `obj_token`：用于 `docs +fetch/update` 读写文档内容
> - `node_token`：用于 `wiki +node-list/create/delete` 管理知识库结构
> - `parent_node_token`：同步回飞书时，确定新文档应放在哪个父节点下

## 同步流程

### 1. 前置条件

```bash
# 安装 lark-cli
npm install -g @larksuite/cli

# 配置应用（首次使用）
lark-cli config init --new

# 授权用户身份（需要 wiki + docs 权限）
lark-cli auth login --domain wiki,docs

# 可用版本：1.0.46（推荐，1.0.59+ 的 +shortcut 有 bug）
npm install -g @larksuite/cli@1.0.46
```

> ⚠️ **注意**：lark-cli v1.0.59+ 的 `+shortcut` 命令存在 bug，会报 `"positional arguments are not supported"`。解决方案：
> 1. 降级到 v1.0.46
> 2. 或通过 Node.js `execFileSync` 直接调用二进制（绕过 shell 参数解析问题）

### 2. 获取知识库节点列表

```bash
# 列出所有知识空间，找到目标 space_id
lark-cli wiki +space-list --as user

# 列出根节点
lark-cli wiki +node-list --as user --space-id <space_id>

# 列出子节点
lark-cli wiki +node-list --as user --space-id <space_id> \
  --parent-node-token <node_token>
```

节点返回结构：
```json
{
  "node_token": "唯一节点token",
  "node_type": "origin",
  "obj_token": "文档内容token",
  "obj_type": "docx",
  "parent_node_token": "父节点token",
  "title": "文档标题",
  "has_child": true
}
```

### 3. 读取文档内容

```bash
# 获取文档内容（Markdown 格式）
lark-cli docs +fetch --as user --doc <obj_token> \
  --doc-format markdown
```

返回的 `content` 是 Markdown 文本，第一行是 `# 标题`，写入本地文件时可根据需要调整。

### 4. 更新文档内容（本地→飞书）

```bash
# 覆盖文档全部内容
lark-cli docs +update --as user --doc <obj_token> \
  --command overwrite --doc-format markdown \
  --content "$(cat local-file.md)"
```

### 5. 创建新文档（本地→飞书）

先在飞书知识库中创建节点：
```bash
# 在指定父节点下创建子节点
lark-cli wiki +node-create --as user --space-id <space_id> \
  --title "文档标题" \
  --parent-node-token <parent_node_token>

# 获取返回的 obj_token，然后写入内容
lark-cli docs +update --as user --doc <new_obj_token> \
  --command overwrite --doc-format markdown \
  --content "$(cat new-file.md)"
```

## 自动同步脚本

### 飞书 → 本地（推荐，以飞书为准）

```javascript
const { execFileSync } = require("child_process");
const path = require("path");
const fs = require("fs");

const BIN = "lark-cli.exe 的完整路径";
const MAPPING = JSON.parse(fs.readFileSync(".feishu-mapping.json", "utf8"));

async function syncFromFeishu() {
  for (const entry of MAPPING.files) {
    const result = execFileSync(BIN, [
      "--as", "user",
      "docs", "+fetch",
      "--doc", entry.obj_token,
      "--doc-format", "markdown"
    ], { encoding: "utf8", timeout: 30000 });

    const data = JSON.parse(result);
    if (!data.ok) continue;

    let content = data.data.document.content;
    content = content.replace(/^# .*\n/, "").trimStart();

    const filePath = path.join(LOCAL_DIR, entry.local);
    fs.mkdirSync(path.dirname(filePath), { recursive: true });
    fs.writeFileSync(filePath, `# ${entry.title}\n\n${content}`, "utf8");
  }
}
```

### 本地 → 飞书（按需使用）

```javascript
function syncToFeishu() {
  for (const entry of MAPPING.files) {
    const filePath = path.join(LOCAL_DIR, entry.local);
    if (!fs.existsSync(filePath)) continue;

    const content = fs.readFileSync(filePath, "utf8");
    execFileSync(BIN, [
      "--as", "user",
      "docs", "+update",
      "--doc", entry.obj_token,
      "--command", "overwrite",
      "--doc-format", "markdown",
      "--content", content
    ], { encoding: "utf8", timeout: 30000 });
  }
}
```

## 关键坑点与注意事项

### 1. lark-cli 版本兼容性

- v1.0.46：`+shortcut` 命令正常工作
- v1.0.56-v1.0.59：`+shortcut` 命令报错（含 `api` 命令）
- 内置子命令（`wiki spaces list`、`wiki nodes list`、`drive files list`）在所有版本都正常
- 通过 Node.js `execFileSync` 调用可绕过 shell 参数解析问题

### 2. token 类型

| token 类型 | 用途 | 获取方式 |
|-----------|------|---------|
| `obj_token` | 读写文档内容（`docs +fetch/update`） | `+node-list` 返回的 `obj_token` |
| `node_token` | 管理知识库结构（`+node-list/create/delete`） | `+node-list` 返回的 `node_token` |
| `parent_node_token` | 确定文档层级关系 | 父节点的 `node_token`，根节点为空字符串 |

### 3. 文档内容格式

- `--doc-format markdown` 返回的 Markdown 去掉了文档标题行（`# 标题`），写入文件时建议手动加上
- 子文档只能用 `node_token` 读取（用 `obj_token` 会报 "Internal error"）
- 根节点的 `parent_node_token` 为空字符串，不是 null

### 4. 认证

```bash
# 首次配置
lark-cli config init --new        # 配置应用凭证
lark-cli auth login --domain wiki,docs  # 用户授权

# 查看当前身份
lark-cli --as user wiki +space-list  # 正常返回 = 用户身份可用
lark-cli --as bot wiki +space-list   # 返回空列表 = bot 无权限
```

### 5. 文件命名建议

- 主文档：保留飞书上的编号前缀（如 `4.掌握Markdown语法.md`）
- 子文档：保留原名，与主文档放在同一文件夹
- 文件夹名：用主文档标题作为文件夹名
- 避免文件名中的特殊字符：`/ \ : * ? " < > |`

## 恢复出厂设置

如果需要重新开始：

```bash
# 完全卸载
npm uninstall -g @larksuite/cli
rm -rf ~/.lark-cli

# 重装
npm install -g @larksuite/cli@1.0.46
lark-cli config init --new
lark-cli auth login --domain wiki,docs
```
