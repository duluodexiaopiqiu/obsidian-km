# 3.0.1. 知识库的设计 —— 文件夹结构怎么搭？

## 先聊一个核心问题：有了双向链接，还需要文件夹吗？

用过 Obsidian 一段时间后，你可能会产生一个疑问：

> Obsidian 最强的不是双向链接吗？那我还需要费心设计文件夹结构吗？

这个问题问得很好。我们先来看 Obsidian 的设计理念。

### Obsidian 的核心理念：双链驱动

Obsidian 的创始人之一（Silver）在介绍这款工具时，反复强调一个观点：

**只要有一个入口笔记，你就可以通过双向链接一直点、一直点、一直点，最终找到你需要的任何内容。**

听起来很美好。从"自己看"的角度，确实不需要层级目录结构：

```
入口笔记（比如"我的知识库总览"）
    ├──→ [[编程学习]]
    │       ├──→ [[Python 基础]]
    │       │       └──→ [[装饰器详解]]
    │       └──→ [[JavaScript]]
    │               └──→ [[闭包与作用域]]
    ├──→ [[2026 年项目]]
    │       ├──→ [[项目A]]
    │       └──→ [[项目B]]
    └──→ [[读书笔记]]
            └──→ [[如何阅读一本书]]
```

你要做的就是从一个入口出发，跟着链接走就行。这也是为什么很多人说"Obsidian 让你不用再纠结分类"。

### 但现实是：我们不能完全依赖双链

把全部希望寄托在双链上，有几个现实问题绕不开：

**1. 给别人看怎么办？**  
你不可能让每个人都装一个 Obsidian。当你需要分享知识库时，对方打开的是一个文件夹，里面一堆 `.md` 文件，如果没有合理的目录结构，对方根本无从下手。

**2. 导出 PDF 怎么办？**  
当你把笔记导出成 PDF 发给别人时，PDF 没有"双向链接"这一说。它是一份线性文档，需要清晰的章节结构。

**3. Obsidian 打不开怎么办？**  
虽然概率很低，但如果哪天 Obsidian 出了什么问题，或者你换工具了，你面对的就是一堆 Markdown 文件。这时文件夹结构就是你的救命稻草。

**4. 大脑不总是知道"入口"在哪**  
双链依赖一个前提：你知道从哪条链开始。但有时候你只是随手翻翻，像翻书一样浏览，这时候文件夹结构比双链更高效。

### 结论

> **好的目录结构 = 不依赖 Obsidian 也能做好知识管理。**

双链是"灵魂"，文件夹是"骨架"。灵魂让知识活起来，骨架让知识站得住。两者缺一不可。

## 知识库文件夹设计的常见思路

下面介绍几种主流的设计思路，你可以根据自己的需求选择或组合。

### 方案一：按领域/主题分类

这是最通用的方式，适合大多数个人知识库。

```
我的知识库/
├── 01-编程开发/
│   ├── Python/
│   ├── JavaScript/
│   └── 工具链/
├── 02-产品设计/
│   ├── 需求分析/
│   └── UI设计/
├── 03-阅读笔记/
│   ├── 技术类/
│   └── 人文类/
├── 04-生活记录/
│   ├── 旅行/
│   └── 健康/
└── 05-随笔想法/
```

**适合人群：** 知识领域比较清晰，且领域之间不常交叉的人。

**优点：** 直观，归档方便。  
**缺点：** 跨领域的内容难以归类（比如"用 Python 做数据分析"该放编程还是数据科学？）。

---

### 方案二：PARA 方法（项目驱动）

著名生产力专家 Tiago Forte 提出的方法，将笔记分为四个核心类别。

```
我的知识库/
├── 1-项目/          ← 有明确截止日期的任务
│   ├── 公司官网改版/
│   └── 博客搭建/
├── 2-领域/          ← 长期关注的领域
│   ├── 编程/
│   ├── 设计/
│   └── 写作/
├── 3-资源/          ← 感兴趣的主题/素材
│   ├── 工具推荐/
│   └── 好文收藏/
└── 4-归档/          ← 已完成/不再活跃的内容
    └── 2024项目/
```

**适用人群：** 工作中需要频繁切换项目的人，或者想用知识库来驱动行动的人。

**优点：** 以行动为导向，项目完成后直接归档，结构整洁。  
**缺点：** 对纯学习型知识库来说，"领域"和"资源"的界限有时比较模糊。

---

### 方案三：按生命周期/状态分类

**这套方案我在实际用的，也是这套教程的实战案例。**

核心思想是：**"我的飞书"就像一台内容工厂**，内容是有生命周期的——从灵感到成品，再到分发和复盘。

```
我的飞书/                          ← 知识库根目录（Vault）
├── 0-收件箱/                     ← 灵感碎片、待处理内容
│   ├── 截图/
│   └── 闪念笔记/
├── 1-项目/                       ← 正在进行的项目
│   ├── 封面设计/
│   └── 视频脚本/
├── 2-领域/                       ← 长期沉淀的知识领域
│   ├── 编程/
│   └── 写作/
├── 3-知识库内容/                 ← 输出的成品/文章
│   ├── Obsidian教程/
│   └── 工具测评/
├── 4-归档/                       ← 已完成的项目
│   └── 2025/
└── 5-模板/                       ← 笔记模板
```

**适合人群：** 内容创作者、有输出需求的人。

**优点：**

- 跟内容生产的实际流程吻合
- 收件箱让你随时捕捉灵感，不用纠结放哪
- 项目完成后归档，保持主目录整洁

---

### 方案四：按时间线分类

适合日记、周报、项目日志等时间敏感的内容。

```
我的知识库/
├── 日记/
│   ├── 2026/
│   │   ├── 01-一月/
│   │   └── 02-二月/
│   └── 2025/
├── 周报/
└── 项目日志/
```

**适合人群：** 习惯写日记、有定期复盘习惯的人。

**优点：** 按时间回溯非常方便。  
**缺点：** 单靠时间难以组织知识内容。

## 几条通用的设计原则

不管选哪种方案，这几条原则都适用：

### 1. 文件夹不超过 3 层

```
❌ 编程/后端/Python/框架/Django/教程
✅ 编程/Python/Django
```

太深的层级会让你找笔记时点半天，而且 Obsidian 的侧边栏也显示不全。

### 2. 用数字前缀控制排序

文件夹默认按字母排序，用数字前缀可以自定义顺序：

```
❌ 项目  领域  归档  模板  收件箱
✅ 0-收件箱  1-项目  2-领域  3-归档  9-模板
```

这样"收件箱"永远排在最前面，"模板"放在最后。

### 3. 收件箱（Inbox）是必需品

不管你的结构多完美，总有一些笔记是突然冒出来的。有一个"收件箱"文件夹，你就能随时扔进去，之后再整理。

**不要让"不知道该放哪"成为记笔记的阻碍。**

### 4. 入口笔记（MOC）是文件夹的补充

MOC（Map of Content，内容地图）是一篇笔记，里面列出某个主题下的所有相关笔记链接。

比如你有一个 `编程/Python/` 文件夹，里面放了很多 Python 笔记。你可以在文件夹根目录放一个 `Python MOC.md`，里面用链接组织这些笔记：

```markdown
# Python 学习地图

## 基础
- [[Python 安装与环境配置]]
- [[变量与数据类型]]
- [[流程控制]]

## 进阶
- [[装饰器]]
- [[生成器]]
- [[面向对象编程]]

## 实战
- [[用 Python 做数据分析]]
- [[Flask Web 开发]]
```

这样即使没有文件夹，你也能通过 MOC 找到一切。

### 5. 定期回顾和调整

没有哪个目录结构是一开始就完美的。用几个月后，你会发现某些分类不合理，某些文件夹过多了——**大胆调整**。知识库是为你的思考服务的，不是为某个完美的"分类学"服务的。

## 实战：这套教程的知识库结构

回到我们这套教程，我自己的"我的飞书"知识库就是按"生命周期"来设计的：

```
📁 我的飞书/
├── 📁 0-收件箱/          ← 突然冒出的想法，先丢这里
├── 📁 1-项目/            ← 正在做的事
│   ├── Obsidian教程
│   └── 封面改版
├── 📁 2-领域/            ← 长期积累的知识
│   ├── 编程
│   ├── 自媒体
│   └── 写作
├── 📁 3-知识库内容/      ← 输出的成品
├── 📁 4-归档/            ← 已完成的项目
└── 📁 9-模板/            ← 笔记模板
```

而你现在正在看的这篇文章，就在 `3-知识库内容/Obsidian教程/` 下面。

这个结构的优点是：

- **收件箱**保证了捕捉速度，不用纠结分类
- **项目**驱动行动，有明确的输出目标
- **知识库内容**是沉淀的成果，做完了就是一篇好文章
- **归档**让主目录保持清爽

## 给你的建议

如果你刚开始搭建知识库，不要追求一步到位：

1. **先用一个简单的结构开始** —— 比如上节课提到的收件箱 + 项目 + 领域
2. **用起来再说** —— 先记 50 篇笔记，你自然会知道哪里不合理
3. **边用边调整** —— 每两个月回顾一次目录结构
4. **记住核心原则** —— 文件夹负责"找到"，双链负责"发现"

> 好的知识库设计，不是让你一次想清楚所有分类，而是让你在知识积累的过程中，始终保持有序。
